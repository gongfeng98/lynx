# iOS Node-API Addons

The open-source Explorer iOS sample includes a `LynxNodeAPI` host module and native loader example for integrating a Node-API addon.

For the experimental capability model, SDK prerequisites, the role of the sample `LynxNodeAPIModule` in Explorer, and addon authoring guidance, see [Lynx Node-API Addons](../../docs/lynx-node-api.md). This document only covers iOS-specific packaging.

## How To Integrate The Addon Into The App

Apple platforms prefer static integration. On iOS, integrate the addon through a `podspec` that packages the addon static library as an `xcframework`. CocoaPods links the library into the app at build time.

The addon generated by `@lynx-js/weak-node-api` also provides an `addon_use.h` header. Include this header from one host translation unit so its `NAPI_USE(<addon_symbol>)` reference keeps the addon's static registration symbol alive during the final app link. The Explorer sample includes `addon_use.h` from `AppDelegate.mm` when the header is available through the addon pod headers. Do not use `-force_load` for this integration path.

## How To Enable It For A Page

The current iOS Explorer sample only enables the Node-API example integration when the page URL query includes `enable_napi_addon=1` (or `enable_napi_addon=true`).

Example:

- `file://lynx?local://homepage.lynx.bundle?enable_napi_addon=1`

## How It Is Resolved

When JS calls `requireNodeAddon("<addon>")`, the loader resolves the addon from the host's static addon registry. The addon name passed from JS must match the generated addon name.
